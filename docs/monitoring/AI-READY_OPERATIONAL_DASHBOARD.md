**版本**: v1.0

## 一、概述

### 1.1 设计目标

- **实时监控**: 每15秒刷新核心指标
- **多维度展示**: 5个核心指标组
- **告警预警**: 三级告警机制（普通/严重/紧急）
- **故障定位**: 快速定位问题模块
- **历史追溯**: 7天数据保留+指标趋势

### 1.2 仪表盘定位

| 属性 | 说明 |
|------|------|
| 使用对象 | 运维工程师、开发人员、技术负责人 |
| 刷新粒度 | 核心指标15秒，详细指标60秒 |
| 访问方式 | Web UI (Dashboard) + API (监控系统集成) |
| 数据源 | Prometheus + 自定义健康检查脚本 |

## 三、核心指标定义

### 3.1 系统级指标

| 指标名称 | 表达式 | 告警阈值 | 说明 |
|----------|--------|----------|------|
| 系统健康状态 | health_score() | >90: healthy, >70: degraded, ≤70: unhealthy | 综合健康评分 |
| CPU使用率 | 100 - (avg(irate(node_cpu_seconds_total{mode="idle"}[1m])) * 100) | >80% warning, >90% critical | CPU整体使用率 |
| 内存使用率 | (1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) * 100 | >85% warning, >90% critical | 内存使用率 |
| 磁盘使用率 | (1 - node_filesystem_avail_bytes / node_filesystem_size_bytes) * 100 | >80% warning, >90% critical | 磁盘使用率 |
| 网络流入 | sum(rate(node_network_receive_bytes_total[1m]))

) | >100MB/s warning | 网络流入带宽 |
| 网络流出 | sum(rate(node_network_transmit_bytes_total[1m]))

) | >100MB/s warning | 网络流出带宽 |

### 3.2 应用级指标

| 指标名称 | 表达式 | 告警阈值 | 说明 |
|----------|--------|----------|------|
| API健康状态 | up{job="ai-ready-api"} | 0 告警 | API服务是否在线 |
| API响应时间 | histogram_quantile(0.95, sum(rate(http_server_requests_seconds_bucket[5m])) by (le)) | >2s warning, >5s critical | P95响应时间 |
| API错误率 | sum(rate(http_server_requests_seconds_count{status=~"5.."}[1m])) / sum(rate(http_server_requests_seconds_count[1m])) | >1% warning, >5% critical | 5xx错误率 |
| 数据库连接数 | pg_stat_activity_count | >15 warning | PostgreSQL连接数 |
| Redis连接数 | redis_connected_clients | >50 warning | Redis客户端连接数 |
| 接口调用量 | sum(rate(ai_ready_api_requests_total[1m])) | - | QPS |
| 缓存命中率 | rate(redis_keyspace_hits_total[1m]) / (rate(redis_keyspace_hits_total[1m]) + rate(redis_keyspace_misses_total[1m])) | <90% warning | Redis缓存命中率 |

### 3.3 业务级指标

| 指标名称 | 说明 | 告警阈值 | 数据源 |
|----------|------|----------|--------|
| 用户登录数 | 每分钟登录用户数 | <100 warning (业务低谷) | 日志分析 |
| 事务处理量 | 每分钟事务数 | - | 业务日志 |
| Agent调用次数 | AI Agent调用计数 | >1000 warning | Agent日志 |

## 四、告警机制设计

### 4.1 告警等级划分

| 等级 | 颜色 | 名称 | 响应时间 | 处理优先级 |
|------|------|------|----------|------------|
| P0 | 红色 | 紧急 | 5分钟 | 立即处理 |
| P1 | 橙色 | 严重 | 30分钟 | 尽快处理 |
| P2 | 黄色 | 普通 | 4小时 | 正常处理 |
| P3 | 绿色 | 提醒 | 24小时 | 计划处理 |

### 4.2 告警规则示例

```yaml
# 告警规则配置
groups:
  - name: ai_ready_alerts
    rules:
      - alert: API服务不可用
        expr: up{job="ai-ready-api"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "AI-Ready API服务不可用"
          description: "API服务连续1分钟无法访问"
      
      - alert: CPU使用率过高
        expr: 100 - (avg(irate(node_cpu_seconds_total{mode="idle"}[1m])) * 100) > 80
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "CPU使用率高于80%"
          description: "当前CPU使用率: {{ $value }}%"
      
      - alert: 数据库连接数过多
        expr: pg_stat_activity_count > 15
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "数据库连接数超过15"
          description: "当前连接数: {{ $value }}"
      
      - alert: 响应时间过长
        expr: histogram_quantile(0.95, sum(rate(http_server_requests_seconds_bucket[5m])) by (le)) > 2
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "P95响应时间超过2秒"
          description: "当前P95响应时间: {{ $value }}s"
```

### 4.3 告警通知渠道

| 渠道 | 触发条件 | 人员 | 策略 |
|------|----------|------|------|
| 飞书机器人 | P0/P1 | 运维团队 | 实时推送 |

## 五、数据刷新和存储策略

### 5.1 数据刷新策略

| 指标类别 | 刷新间隔 | 保留时间 | 查询延迟 |
|----------|----------|----------|----------|
| 核心摘要 | 15秒 | 7天 | <500ms |
| 服务监控 | 30秒 | 7天 | <1s |
| 详细指标 | 60秒 | 30天 | <2s |
| 告警事件 | 实时 | 90天 | <100ms |

### 5.2 数据存储策略

| 数据类型 | 存储系统 | 保留策略 | 查询方式 |
|----------|----------|----------|----------|
| 指标数据 | Prometheus | 30天 | PromQL |
| 告警记录 | MySQL | 90天 | REST API |
| 日志数据 | Elasticsearch | 14天 | Kibana |
| 非关键历史 | TimescaleDB | 1年 | SQL |

### 5.3 性能优化

| 优化项 | 方案 |
|--------|------|
| 查询优化 | 下钻查询 + 数据采样 |
| 缓存策略 | Redis缓存5秒 |
| 数据压缩 | Prometheus配置Snappy压缩 |
| 分片策略 | 按时间分片，每日分割 |

## 六、实现方案

### 6.1 技术栈

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| 前端 | Vue 3 + Element Plus | 响应式仪表盘 |
| 后端 | Spring Boot | REST API服务 |
| 数据源 | Prometheus + MySQL + ES | 指标存储 |
| 安全 | JWT认证 + RBAC | 访问控制 |

### 6.2 访问控制

| 角色 | 权限 |
|------|------|
| 运维工程师 | 完整访问 |
| 开发人员 | 服务查看 + 无告警处理 |
| 技术负责人 | 完整访问 + 报告导出 |
| 管理员 | 全部权限 + 系统配置 |

### 6.3 关键接口

```yaml
# API接口定义
GET  /api/dashboard/summary          # 摘要数据
GET  /api/dashboard/services         # 服务列表
GET  /api/dashboard/metrics?metric=xxx  # 指标详情
GET  /api/dashboard/alerts           # 告警列表
POST /api/dashboard/acknowledge      # 告警确认
GET  /api/dashboard/history?start=xxx&end=xxx  # 历史数据
GET  /api/dashboard/export           # 导出报告
```
